## [FAIL] Scenario 3 — Shop Detail Public Access
**Expected:** GET /shops/{id} returns shop details with photos and taxonomy tags.
**Actual:** GET /api/shops/{id} and GET http://localhost:8000/shops/{id} both return 500.

**Root cause — two bugs:**

Bug 1 — backend/api/shops.py:61: Supabase query requests shop_tags(tag_id, tag_name, ...) but the shop_tags table has no tag_name column (only shop_id, tag_id, confidence, distinctiveness). PostgREST returns 500 on any unknown column.
Fix: Remove tag_name from the select. Tag names come from the joined taxonomy_tags(label, label_zh).

Bug 2 — All 200 shops have slug: null. The backfill script (backend/scripts/backfill_slugs.py) was never run. The /shops/[shopId]/[slug] route cannot be tested or shared.
Fix: Run python backend/scripts/backfill_slugs.py against the local DB.

## [FAIL] Scenario 4 — Signup + PDPA Consent
**Expected:** After signup, POST /api/auth/consent succeeds and records consent.
**Actual:** POST /api/auth/consent returns 401.

**Root cause — lib/api/proxy.ts:13-32:**
The proxy builds its own headers from scratch, reads the SSR Supabase session cookie, and injects a JWT if found. It never reads the incoming request's Authorization header. The consent page sends Authorization: Bearer {token} from the browser, but the proxy drops it silently.

Post-signup, there is a timing window before the SSR cookie is established. The proxy sends no JWT during this window — backend returns 401.

Backend consent endpoint works correctly when called directly with a valid JWT (tested — returns 200).

Fix: After the SSR session check, fall back to the incoming Authorization header:
  if (!headers['Authorization']) {
    const incoming = request.headers.get('Authorization');
    if (incoming) headers['Authorization'] = incoming;
  }

Add shop to list FAIL: POST /lists/{id}/shops expects {"shop_id": "..."} (snake_case) but the frontend sends {"shopId": "..."} (camelCase per the API contract) — returns 422. Response is also snake_case (list_id, shop_id, added_at) — inconsistent with the rest of the API.

Root cause: ListItemCreate and ListItem Pydantic models extend plain BaseModel, not CamelModel. They were not updated when the camelCase migration was applied.
Fix: Extend those models with CamelModel and add populate_by_name=True.
